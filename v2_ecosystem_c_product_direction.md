# V2 生态型 C 端产品方向修正

**日期**:2026-05-27  
**性质**:对 `v2_final_report.md` 与 `v2_report_review_and_direction.md` 的方向修正。  
**核心纠偏**:我们不是要把 C 端产品降级成“采数据外壳”,而是要做一个真实有 C 端价值的产品;只是它的第一性商业目的不是 C 端收费,而是通过 C 端产品构建职业数据生态,持续反哺 B 端 AI 招聘产品。

---

## 0. 最核心判断

我们的产品必须是 **真正的 To C 产品**,而不是伪装成 To C 的数据采集器。

但它也不是普通 consumer SaaS。它的目的不是靠 C 端订阅赚钱,也不是做内容社区,更不是做新的 LinkedIn。

它的第一性目标是:

> 给 C 端用户提供足够明确、足够持续、足够可信的职业利好,让他们愿意自愿、长期、高质量地贡献自己的简历、技能、能力证据、职业偏好和机会状态,从而为 B 端 AI 招聘产品建立数据生态。

所以这件事的本质是:

> **To C 产品形态 + B 端数据生态目的。**

这和“只做数据钱包”不同。数据钱包是内部能力,不是用户购买理由。用户购买或使用的理由必须是对他自己有利。

---

## 1. 第一性目的重新表述

### 1.1 公司层目的

公司不是为了做一个 C 端小工具而做 C 端。

公司要的是:

1. 打通 C 端人才生态。
2. 获得比 LinkedIn 更新、更细、更真实的职业数据。
3. 让用户自己持续更新这些数据。
4. 最终让 B 端 AI 招聘产品推荐更准、触达更强、护城河更深。

### 1.2 C 端产品层目的

C 端产品必须回答用户的问题:

> 我为什么要来?  
> 我为什么要填?  
> 我为什么要连账号?  
> 我为什么要持续更新?  
> 我为什么相信你不会把我卖掉?

如果回答不了这些问题,就算 B 端很想要数据,C 端也不会持续供给。

因此,C 端产品必须是一个 **利他性伪装不足、利己性真实存在** 的产品。它不能只说“帮你找工作”,也不能只说“帮我们优化匹配”。它要真的让用户觉得:

- 我更了解自己的能力。
- 我更容易整理自己的职业资产。
- 我更容易在合适的时候被看到。
- 我可以掌控自己是否开放机会。
- 我得到的东西大于我交出的数据。

---

## 2. 不是“C 端壳”,而是“生态入口”

前一版 review 中“能力镜子是交互外壳”的表述容易误导。更准确应改成:

> 能力镜子是 C 端产品的真实价值入口,也是生态数据循环的前台界面。

它不是假壳。用户必须真的有用。

但它也不是终点。它必须把用户行为沉淀成结构化数据资产。

### 2.1 正确关系

```
C 端用户利好
  ↓
用户愿意持续使用
  ↓
用户愿意授权 / 填写 / 更新数据
  ↓
形成结构化职业数据资产
  ↓
反哺 B 端推荐、匹配、触达
  ↓
B 端价值增强
  ↓
反过来给 C 端用户更好的机会反馈
```

这才是生态闭环。

### 2.2 错误关系

```
想要 B 端数据
  ↓
做一个 C 端页面让用户填资料
  ↓
用户发现自己被采集
  ↓
不更新 / 不信任 / 流失
```

这条路不能走。

---

## 3. C 端必须给用户什么利好

即使免费,产品也必须让用户感到收益明确。建议把 C 端利好分成 6 类。

### 3.1 职业资产整理

用户长期痛点是职业资料分散:

- LinkedIn 上有经历。
- 简历里有旧版本。
- GitHub / Behance / Substack / 个人站上有作品。
- Notion / Google Docs 里有项目沉淀。
- Slack / CRM / HubSpot / Clay / Figma 里有过程证据。

产品可以帮用户做一件很朴素但高频的事:

> 自动整理我的职业资产,让我不用每次求职、接项目、申请合作时从零开始翻材料。

这是 C 端第一利好。

### 3.2 能力画像和技能地图

用户不是每天都在找工作,但会关心自己有没有成长、是否被时代甩开、AI 能力到底算不算优势。

产品可以给用户:

- 当前能力地图
- 技能变化趋势
- AI 工具栈分析
- 可迁移能力识别
- 与目标岗位的差距
- 哪些能力缺少证据

这就是“能力镜子”的核心价值。

### 3.3 简历和 Profile 自动更新

很多人不更新简历,不是不需要,而是麻烦。

产品可以提供:

- 自动从项目和平台活动中更新简历
- 自动生成不同版本 resume
- 自动生成 LinkedIn About / Featured 内容
- 自动生成项目描述
- 自动生成求职或合作时的 one-page profile

这直接服务用户,也直接获取高质量简历数据。

### 3.4 能力证据和可信背书

AI 时代最大问题是 claim 变便宜。人人都能说自己会 AI、会自动化、会增长、会策略。

产品给用户的利好是:

> 不是帮你吹,而是帮你证明。

具体形式:

- evidence card
- workflow case study
- 项目结果证明
- 数据来源时间戳
- 跨平台一致性
- AI 参与说明
- 可分享 proof page

这对用户有利,也对 B 端有利。

### 3.5 私密机会控制

大量用户愿意被更好机会发现,但不愿公开求职。

产品可以提供一个低风险机制:

- 默认不公开求职。
- 用户设置机会偏好。
- 只有匹配到明确机会才提醒。
- 用户先审查机会。
- 用户同意后才把资料给 B 端。

这比 LinkedIn Open-to-Work 更安全,也更适合 passive candidates。

### 3.6 职业生态反馈

如果平台有足够 B 端需求,还可以反向给 C 端反馈:

- 哪些技能正在被企业搜索
- 哪些 AI 工具栈更值钱
- 哪些岗位开始要求某类能力
- 用户当前 profile 缺什么证据
- 用户适合哪些转型方向

这会形成生态飞轮:越多 B 端需求,越能给 C 端更准确反馈;越多 C 端更新,越能给 B 端更好数据。

---

## 4. 免费也可以,但不能没有交换逻辑

C 端不收费完全可以。甚至早期建议免费。

因为真正的商业价值在 B 端数据生态,不是 C 端订阅收入。

但免费不等于没有交换。用户仍然要明白:

> 我给你数据,你给我职业收益。

这种收益可以不是钱,而是:

- 更好的简历
- 更清晰的能力地图
- 更可信的能力证明
- 更少求职焦虑
- 更安全的被发现方式
- 更精准的机会
- 更及时的职业趋势反馈

只要用户觉得交换公平,他就不会觉得被利用。

## 5. 产品定位建议

### 5.1 对外定位

不要说:

- 我们是招聘平台。
- 我们是 LinkedIn 替代品。
- 我们帮你把资料给企业看。
- 我们是职业数据钱包。

建议说:

> 一个帮你整理、证明和更新职业能力的 AI career workspace。

或者:

> 把你的简历、项目、作品和 AI 工作流整理成一份持续更新的能力档案。默认私密,需要时再分享。

更口语一点:

> 你的职业能力自动档案。平时帮你整理自己,机会来时帮你证明自己。

### 5.2 对内定位

内部必须说得更直接:

> 这是 B 端 AI 招聘产品的数据生态入口。C 端产品的一切体验,都要服务于用户自愿、持续、高质量地更新职业数据。

## 6. MVP 应重新设计为“C 端利好驱动的数据采集”

### 6.1 MVP 主路径

用户进入产品后,主路径应是:

1. 导入现有简历 / LinkedIn export / 手动 profile。
2. 连接 1 个职业证据源,如 GitHub、Substack、个人站、Behance、Dribbble、Notion。
3. AI 生成初版职业能力档案。
4. 用户确认和修正。
5. 系统指出能力缺口和证据缺口。
6. 用户生成 1 张能力证据卡。
7. 用户选择是否公开、分享或保持私密。
8. 用户设置是否接收机会提醒。

这个路径同时满足 C 端利好和 B 端数据目标。

### 6.2 MVP 第一屏不要过度强调“连接账号”

“连接账号”对用户是风险动作,不是自然动作。

更好的第一屏是:

> 上传你的简历或粘贴 LinkedIn 链接,30 秒生成你的 AI 能力档案。你可以选择继续连接作品源,让档案更可信。

这样顺序更自然:

- 先给低风险输入。
- 再给用户看初始价值。
- 再引导高信任授权。

### 6.3 MVP 必须采集的最小数据

即使产品免费,也必须获取以下核心数据,否则无法反哺 B 端:

1. 当前职位
2. 当前公司
3. 工作年限
4. 地区
5. 邮箱或可联系渠道
6. 技能标签
7. AI 工具栈
8. 项目 / 作品 / 案例
9. 来源链接或证明
10. 机会开放状态
11. 目标岗位或偏好方向
12. 最近更新时间

没有这些字段,产品就没有招聘数据价值。

## 7. 生态飞轮

### 7.1 第一阶段:用户整理自己

用户来是为了整理简历、项目、技能、AI 能力。

平台得到基础 profile、技能、项目、证据。

### 7.2 第二阶段:用户证明自己

用户生成 evidence card、proof page、能力档案。

平台得到高维能力证据和真实性信号。

### 7.3 第三阶段:用户控制机会

用户设置机会偏好和开放程度。

平台得到 passive candidate 的隐性求职 / 可挖动信号。

### 7.4 第四阶段:B 端反向反馈

B 端需求进入系统。

平台告诉用户哪些能力被需要、哪些证据要补、哪些机会适合。

### 7.5 第五阶段:生态增强

C 端数据越多,B 端匹配越准。  
B 端需求越多,C 端越愿意更新。  
这就是生态。

---

## 8. 应避免的误区

### 8.1 不要把 C 端做成问卷

如果用户感觉自己在帮平台填数据库,产品会死。

### 8.2 不要只做漂亮页面

漂亮页面不能形成长期数据更新。

### 8.3 不要过早做社交

社交会引入内容焦虑、冷启动难题和 LinkedIn 阴影。

### 8.4 不要把付费作为第一目标

早期 C 端收费会降低数据规模。只要 B 端数据价值明确,免费是合理策略。

### 8.5 不要隐藏 B 端目的

不能在产品首页大喊“我们要把你推荐给企业”,但也不能欺骗用户。正确方式是透明控制:

> 默认私密。你可以选择是否让机会方看到你。你可以随时关闭。

---

## 9. 重新定义 90 天验证目标

90 天不是验证“用户会不会为能力镜子付费”。

真正要验证的是:

1. 用户是否愿意导入简历 / profile。
2. 用户是否愿意让 AI 结构化自己的技能。
3. 用户是否愿意补充和确认技能。
4. 用户是否愿意连接至少一个证据源。
5. 用户是否愿意生成能力证据卡。
6. 用户是否愿意填写机会偏好。
7. 用户是否愿意在 30 天内回来更新。
8. B 端是否认为这些数据比 LinkedIn profile 更有用。

建议 90 天硬指标:

| 指标 | 合格线 |
|---|---:|
| 100 个测试用户完成基础 profile | 60 |
| 导入简历 / LinkedIn export / 手动经历 | 50 |
| AI 抽取技能后用户确认 | 45 |
| 连接外部证据源 | 35 |
| 生成 evidence card | 30 |
| 填写机会偏好 | 25 |
| 第 30 天仍有更新行为 | 20 |
| B 端访谈认为数据有增量价值 | 5 个中 3 个 |

---

## 10. 最终方向建议

当前方向应从:

> 职业能力镜子

升级为:

> 生态型 C 端职业产品:以用户职业利好为入口,以结构化职业数据为资产,以 B 端机会反馈为飞轮。

具体产品可以叫 Career Mirror,但内部战略应叫:

> C-side Career Data Ecosystem for AI Recruiting。

最重要的一句话:

> C 端产品必须真实利好用户,否则没有持续数据;但它的根本商业目的不是赚 C 端的钱,而是用真实 C 端价值换取用户持续授权,构建 B 端无法从 LinkedIn 或简历库获得的高质量、实时、多维职业数据生态。
