# V2 产品设计复盘：从“大而全职业系统”回到“职业行动社群”

**日期**：2026-05-27  
**项目**：Metix.ai C 端候选人生态产品  
**背景**：本轮设计从“AI 时代职业探索、技能树、行动证明、全球可见性”展开，经过产品负责人反馈后，需要复盘方案中不成熟的地方，并沉淀下一轮设计原则。

---

## 0. 复盘结论

本轮方案不是完全错误，核心判断中有一个方向是成立的：

> AI 时代，简历不足以证明一个人。真实行动、项目片段、workflow、作品、观点实验和他人反馈，会成为更重要的人才信号。

但问题在于，我们把这个强方向扩展成了过大的系统：

```text
职业雷达 + 技能树 + 知识推荐 + 账号连接 + 能力图谱 + 成长打卡 + 个人主页 + 全球可见性
```

这导致产品从“证明自己”偏移到了“知识管理 / 职业管理 / 个人主页 / 成长系统”。逻辑上完整，但用户动机不够尖锐，离求职和 B 端招聘价值也变远。

产品负责人反馈中的核心信号可以理解为：

> 证明自己的方向是好的；但关注、技能、知识管理、静态个人主页都不是足够强的 C 端吸引力。真正需要强调的是社群感、动态参与、反馈、被看见和真实行动。

下一轮应该收敛为：

> **Metix.ai C 端：AI 时代职业行动社群。用户围绕真实 AI 工作流、项目片段、作品和想法提交 Action Proof，在社群反馈中被看见，并把动态能力证据反哺 Metix.ai B 端招聘生态。**

---

## 1. 本轮设计的正确部分

### 1.1 “证明自己”是正确方向

简历和静态 profile 在 AI 时代会变得不够充分。原因是：

- AI 降低了跨领域学习和工具使用门槛。
- 候选人可以更快尝试新行业、新岗位和新 workflow。
- 静态经历无法充分表达一个人的学习速度、行动力、判断力和工具整合能力。
- B 端 recruiter 需要更真实的能力信号，而不是只有自我描述。

因此，Action Proof 的方向是值得保留的。

Action Proof 应该表达：

```text
我观察到什么问题
我用了什么工具
我做了什么行动
人的判断在哪里
产出是什么
有什么结果或反馈
谁看过、认可过、收藏过、复用过
```

这比“我掌握某技能”更接近真实招聘判断。

### 1.2 B 端生态补全是正确第一性目的

项目的内部第一性目的仍然成立：

> 通过 C 端产品持续获取候选人的能力、技能、简历反馈、行动证明、机会偏好和可见性授权，反哺 Metix.ai / openjobs-recruiter 的候选人搜索、匹配和触达。

但这只是内部目的。它不能直接成为 C 端用户使用理由。

C 端用户需要的表层价值应该是：

- 看见别人怎么做。
- 展示自己正在做什么。
- 获得反馈和认可。
- 有机会被 recruiter 或行业人看见。
- 把真实行动沉淀成可展示的能力证明。

### 1.3 动态证据比静态主页更有价值

个人主页可以作为结果页存在，但不应该成为产品核心。

更重要的是动态行为：

- 提交一个 workflow。
- 参与一个主题挑战。
- 发布一个项目片段。
- 获得同行评论。
- 被精选、收藏、复用。
- 被机会方请求查看。

因此，主页应是动态 proof 的沉淀，而不是让用户再维护一个静态职业页面。

## 2. 本轮设计违反的产品设计原则

### 2.1 没有先锁定一个强动机核心行为

本轮设计里有很多行为：

```text
看新闻
关注领域
点亮技能
连接 GitHub
生成 proof
更新简历
看知识推荐
成长打卡
开放机会
维护主页
```

这些都能讲通，但没有先证明哪一个行为是用户最愿意反复做的。

更成熟的做法应该先压缩成一个验证问题：

> 用户是否愿意在一个垂直 AI 职业社群里，每周提交一个真实 workflow / 项目片段 / 想法证明，并获得反馈？

如果这个行为成立，才有必要继续扩展技能树、能力图谱、成长打卡和全球可见性。

参考原则：Lean Startup 强调 MVP 应以最小努力获得最大量 validated learning，用于验证关键商业假设，而不是一开始构建完整产品。

### 2.2 把内部商业目的过早暴露成 C 端叙事

“补全候选人数据库”是内部目标，不是用户卖点。

用户不会因为平台需要候选人数据而来。用户愿意来的理由是：

```text
我能被看见
我能获得反馈
我能看到别人怎么做
我能证明自己不是只会写简历
我能接触到真实机会
```

本轮方案一直强调 B 端生态目的，但对 C 端即时收益的设计不够强。

### 2.3 出现 feature creep / scope creep

本轮方案不断增加功能：

```text
职业雷达
技能树
账号连接
能力图谱
知识推荐
成长打卡
个人主页
全球机会
B 端闭环
```

每个功能都有道理，但叠加后造成产品重心发散。

Feature creep 的典型风险是：产品因不断增加功能而变得复杂、臃肿、交付变慢，甚至偏离基本用途。

本轮方案的表现是：

- 概念越来越完整。
- 用户路径越来越长。
- 主行为越来越不清楚。
- 汇报文件越来越像系统架构，而不是 C 端产品。

### 2.4 把知识管理误当成用户主需求

“关注 AI 新闻”“技能树”“知识推荐”并不是完全没价值，但它们更接近知识库和学习管理。

问题是：

- 知识管理离求职和招聘较远。
- 用户愿意看，但不一定愿意持续用。
- 它很容易和已有 newsletter、AI 信息流、课程平台、Notion 知识库同质化。
- 它产生的 B 端招聘价值较弱。

因此，知识和技能系统应该退到后台：

```text
职业雷达 = 社群主题来源
技能树 = Proof 分类和能力映射
知识推荐 = 辅助用户补 Proof 缺口
```

而不应该成为主产品。

### 2.5 把静态个人主页想得过重

个人主页不是强产品入口。

用户维护个人主页的频率很低，大多数主页最后会变成“死页面”。

更合理的逻辑是：

```text
社群行动 → 动态 proof → 反馈 / 认可 / 收藏 → 自动沉淀为个人展示页
```

即个人主页是动态行为的结果，不是产品主场景。

### 2.6 缺少社群反馈机制

Proof 如果只是用户自己生成、自己保存，动机较弱。

Proof 如果能被同行看见、评论、收藏、复用、推荐，动机明显增强。

本轮方案缺少这些机制：

- 谁会看到用户提交的 proof？
- 谁会反馈？
- 用户为什么在意反馈？
- 哪些 proof 会被精选？
- B 端什么时候出现？
- 社群身份如何形成？

这也是产品负责人强调“社群感”的原因。

### 2.7 没有明确冷启动路径

如果要做社群，必须回答冷启动问题：

```text
第一个圈子是什么？
第一批用户是谁？
第一批内容从哪里来？
第一周大家提交什么？
谁来评论和精选？
B 端机会何时进入？
```

本轮方案仍偏系统设计，没有定义第一个可执行场景。

## 3. 产品负责人的设计思想

产品负责人的反馈可以总结为三个判断。

### 3.1 C 端产品必须有社群场景

用户不是因为“管理自己”而持续回来，而是因为：

- 看见别人怎么做。
- 参与一个主题。
- 获得反馈。
- 被认可。
- 被比较。
- 被机会方看见。

工具解决整理问题，社群解决动机问题。

### 3.2 不要做另一个静态个人主页

静态主页吸引力弱，用户需求表达很强，但真实使用很低。

更有效的是动态社群行为：

```text
本周提交了什么
获得了谁的反馈
被谁收藏
参与了什么挑战
被哪个机会匹配
```

### 3.3 证明能力要放在互动场景里

Action Proof 的最佳位置不是“生成卡片”，而是：

```text
主题挑战
  ↓
用户提交作品 / workflow / 想法
  ↓
同行反馈
  ↓
优秀案例精选
  ↓
系统沉淀 Evidence Graph
  ↓
B 端看到更真实的人才信号
```

## 4. 下一轮产品方向建议

### 4.1 新一句话定位

> **Metix.ai C 端是 AI 时代职业行动社群：用户通过真实作品、workflow、想法和项目证明自己，并在社群反馈中被机会看见。**

### 4.2 新主线

```text
AI 职业主题圈子
  ↓
每周一个真实工作流 / 项目 / 想法挑战
  ↓
用户提交 proof
  ↓
同行反馈、收藏、精选
  ↓
系统沉淀能力证据
  ↓
B 端 recruiter 看到更真实的人才信号
```

### 4.3 原有模块的新位置

| 原模块 | 新定位 |
|---|---|
| Career Radar | 社群话题来源，决定本周讨论什么、挑战什么 |
| Skill Tree | Proof 分类和能力映射后台，不作为核心前台体验 |
| 账号连接 | Proof 可信度增强，用于减少手动填写 |
| 能力图谱 | B 端候选人理解层和 C 端成长反馈层 |
| 成长打卡 | 社群参与与 proof 更新机制 |
| 个人主页 | 动态 proof 的沉淀页，而不是产品核心 |
| Global Visibility | 用户授权后连接 recruiter 和具体机会 |

### 4.4 第一批可验证场景

建议先选一个垂直社群，而不是全行业铺开。

候选方向：

1. AI Marketing / GTM Ops
2. AI Design / Creative Workflow
3. AI Operations / Automation

建议优先从 **AI Marketing / GTM Ops** 开始，因为它更容易产生非工程类 workflow proof，也更适合展示 AI 时代跨领域机会。

第一周挑战示例：

> 本周主题：AI Lead Enrichment Workflow  
> 提交内容：一个脱敏线索补全 workflow、工具链、人的判断节点、样例输出和反思。  
> 社群反馈：同行评论、收藏、复用、精选案例。  
> B 端信号：recruiter 可看到优秀 proof，但用户授权后才开放更多资料。

## 5. 下一轮验证问题

不要再先问“系统应该有哪些功能”，而要问：

1. 用户是否愿意提交一个真实 workflow / 项目片段 / 想法实验？
2. 用户是否愿意让同行看到并评论？
3. 用户是否愿意持续参与每周主题？
4. 用户是否在意被精选、被收藏、被 recruiter 看见？
5. B 端 recruiter 是否认为这些 proof 比简历更有帮助？
6. 哪类 proof 最能帮助判断候选人真实能力？
7. 哪个垂直方向最容易冷启动？

## 6. 下一轮 MVP 建议

### 6.1 不要先做完整平台

先做一个轻量验证：

```text
一个垂直圈子
三个主题挑战
二十个种子用户
人工精选内容
人工收集反馈
人工向 B 端展示 proof
```

### 6.2 最小产品形态

MVP 可以很轻：

- 一个 landing page。
- 一个社群频道。
- 一个 proof 提交表单。
- 一个每周精选页面。
- 一组 recruiter 反馈访谈。

不需要一开始做完整技能树、能力图谱、账号 OAuth、个人主页和推荐系统。

### 6.3 核心指标

C 端：

- 提交率：看到主题后，有多少人提交 proof？
- 复访率：第二周是否继续参与？
- 互动率：评论、收藏、复用、分享。
- 授权意愿：是否愿意在具体机会下开放资料？

B 端：

- recruiter 是否愿意看 proof？
- proof 是否帮助判断候选人？
- 是否比简历多提供有效信号？
- 是否愿意收藏或触达该用户？

---

## 7. 可复用经验总结

1. 不要把内部商业目标直接当作用户价值。
2. 不要先设计完整系统，要先验证一个核心行为。
3. 不要把“用户说有需求”当成“用户真的会用”。
4. 不要让功能之间只有逻辑关系，没有使用关系。
5. 不要过早把产品做成知识库和知识管理。
6. 不要把静态个人主页当成 C 端核心吸引力。
7. 不要用完整性代替锋利度。
8. Action Proof 是强方向，但必须放进社群反馈和真实机会场景里。
9. 职业雷达和技能树可以保留，但应作为社群内容和 proof 结构的支撑层。
10. 下一轮要先验证“用户是否愿意提交真实行动证明并获得反馈”。
