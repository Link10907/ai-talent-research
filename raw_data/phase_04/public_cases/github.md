# GitHub — Default Public Code Repository

## Source date range: 2024–2026

## Scale
- 100M+ developers (2023, GitHub official)
- 630M total repositories (395M public + ~235M private)
- 121M new repositories created in 2025
- Acquired by Microsoft: $7.5B (June 2018)

## Revenue
- $2B ARR (Latka, 2024 est.)
- GitHub Copilot: 4.7M paid subscribers (Jan 2026); 20M total users (including free tier)
- Copilot revenue alone exceeds GitHub's entire valuation at time of acquisition (Satya Nadella statement)
- GitHub Copilot crosses 20M all-time users (TechCrunch, July 2025)

## Public vs Private Repo Split
- 395M public repos (as of 2025 data) out of 630M total
- ~63% of repos are public; ~37% are private
- Historical note: before 2019, private repos required paid plan; free private repos launched Jan 2019
- Private repos = $4/month (Teams) or free for individuals (post-2019)

## Monetization Path
- Individual: Free (unlimited public/private) → Pro ($4/mo) → no strong incentive to pay individually
- Teams: $4/user/month (private org repos, CI minutes)
- Enterprise: $21/user/month (SAML SSO, advanced security)
- GitHub Copilot: $10 (individual) / $19 (Business) / $39 (Enterprise) per user/month
- GitHub Actions: compute minutes (primary secondary revenue)
- GitHub Marketplace: 30% commission on app sales

## % Users Public
- Developer default: public repos for open-source; private for work/proprietary
- Active OSS contributors: ~40M estimated
- Fortune 100: 90% use GitHub Copilot (GitHub claim, 2025)

## D30 / D90 Retention
- Unknown officially
- Very high stickiness: code history + CI/CD integration + team dependencies = near-impossible to migrate
- Enterprise churn extremely low; individual developer churn also low due to portfolio lock-in

## Design Pattern We Could Copy
- Public work (commits, PRs, issues) = public professional portfolio → B2B talent signal
- Microsoft charges enterprises for Copilot to analyze the same code developers freely contribute
- Equivalent for AI talent: public project work → employer intelligence product

## Fit with "Feed B-Side Data" Goal
- VERY HIGH fit: this is the model most directly applicable
- User generates public work artifacts → platform monetizes those signals via enterprise product
- Key tension: developers aware of this but accept it as norm (Linux kernel/open-source ethos)
