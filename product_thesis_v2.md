## 一、出发点(Why)

### 1.1 核心诉求(必须始终对齐的最深目标)

**唯一最核心目的**:获取候选人的简历或职业产出信息,为现有 B 端 AI 招聘产品建立
**数据级别的护城河**。

这是必须始终回到的锚点。所有产品决策都要回答:"这个决策对核心目的的贡献是
什么?"

### 1.2 数据护城河的形式化定义

```
数据护城河价值 = 数据维度 × 数据规模 × 数据新鲜度 × 数据真实性
```

四个因子任何一个 ≈ 0,整个护城河失效。基于 v1 调研的诚实判断:

| 因子 | LinkedIn 的位置 | 我们的窗口 |
|------|--------------|-----------|
| **数据规模** | 1B+ 用户,垄断 | **0 — 我们永远输** |
| **数据真实性** | Verified Applicant Spotlight + 55M verified | **0 — LinkedIn 主动占领中** |
| **数据新鲜度** | employment-history 模式,1-2 年更新一次 | **有窗口 — LinkedIn 结构性弱** |
| **数据维度** | self-reported 履历 + 单维度 claim | **有窗口 — LinkedIn 结构性缺** |

**结论(置信度:高)**:我们的护城河必须建在**新鲜度 + 维度**两个因子上。规模我们
永远输,真实性 LinkedIn 在主动锁定。

### 1.3 LinkedIn 结构性收不到、但 B 端急需的数据维度

(基于 v1 Phase 1-4 全部综合)

| 数据维度 | 为什么 LinkedIn 收不到 | B 端付费意愿 | v1 引用 |
|---------|---------------------|-----------|--------|
| **能力证据**(proof,而非 claim) | LinkedIn 是 self-reported form 模式 | 极高 | Phase 4 验证 |
| **AI 工作流指纹**(用什么模型/工具/prompt 模式) | LinkedIn 没这个 schema | 高 | Phase 3 全部 |
| **跨平台产出聚合**(GitHub+X+Substack+网站) | LinkedIn 不愿打通竞品 | 高 | Phase 1/4 |
| **"非求职状态"的活跃度** | LinkedIn 的 employment-status 模式不兼容 | 极高(passive candidate 是最贵的) | Phase 1 |
| **职业可挖动性的隐性信号** | LinkedIn 的 Open to Work 是污染性公开信号 | 极高 | Phase 4 |

**这五个维度的并集,就是我们要建的数据护城河。**

### 1.4 从 v1 学到的硬约束(不可妥协)

1. **不能让用户感觉被利用**——否则不会自愿持续提供数据(CONTEXT.md 约束 1)
2. **不能和 LinkedIn 直接抗衡**——用户迁移意愿 2.5/10,3 个替代品 14 个月连续死亡(Phase 2/4)
3. **B 端兑现可存在,但不能是核心卖点**——否则 C 端死(CONTEXT.md 约束 3)
4. **必须在 LinkedIn 5 年窗口期内成形**——LinkedIn 12 个 AI 功能、Verified Applicant 已经在主动锁定(Phase 4)

### 1.5 与 LinkedIn 的关系定义(战略层)

**不是替代,是寄生 + 互补。**

- Day 1 起就支持"导出到 LinkedIn"、"嵌入 LinkedIn 个人页"——让 LinkedIn 视我们为
  能力补充,不是竞争对手
- LinkedIn 不会反击寄生者,只会反击替代者(参考 Bento.me 被 Linktree 收购,
  Read.cv 被 Perplexity 收购——LinkedIn 至今没收购过)
- 用户体验上,我们填补 LinkedIn 的两个洞:"我没地方沉淀能力证据"和"LinkedIn
  让我焦虑,我需要一个能力工具"

## 二、目标(What)

### 2.1 C 端目标(产品交付的)

**让用户用一个"为自己服务"的工具,自愿、持续、高质量地把分散的职业产出聚合起来。**

不是"建立职业品牌"。不是"找到工作"。是"了解和提升自己"。

这个语言区别极其重要。前两个会触发"被审判焦虑";后一个触发"对自己的好奇"。

### 2.2 B 端目标(隐性沉淀的)

**沉淀 LinkedIn 结构性拿不到的五个数据维度**(1.3 节列出),且数据由用户**主动授权**
而非被动采集——这一点决定了未来不会触发数据合规危机。

数据的 B 端货币化**在用户毫不察觉、毫不反感的前提下**完成。

### 2.3 分阶段目标和时间表

| 阶段 | 时间 | C 端目标 | B 端目标(隐性) | 关键 KPI |
|------|-----|---------|--------------|---------|
| **阶段一** | 0-12 月 | 在 1-3 个细分人群里做到"周活专业工具" | **数据沉淀,完全不变现** | 1-10 万 MAU,WAU/MAU ≥ 50%,平均链接账号数 ≥ 3 |
| **阶段二** | 12-24 月 | 用户可选择性公开"能力 page",支持嵌入个人站 | **被动 LLM 索引,仍不卖给 B 端** | 30% 用户公开过 page,page 在 ChatGPT/Perplexity 可被检索 |
| **阶段三** | 24-36 月 | 用户 opt-in "opportunity 通道" | **B 端通道开启,reverse model** | 20% 用户开通 opportunity,B 端 ARR > X |

**关键转折点判定标准**:
- 阶段一→二:用户量 ≥ 10 万 + 平均周活率 ≥ 30%(证明产品本身有粘性,不依赖 B 端兑现)
- 阶段二→三:有至少 10 个**非招聘场景**的成功用例(用户被合作方/投资人/同行找到)——证明产品有"非求职"的网络效应,这时候开 B 端通道不会污染产品

### 2.4 不追求的目标(明确说出来)

- 不追求 LinkedIn 体量(永远做不到,也不必)
- 不追求"职业社交网络"(社交是奖励不是门槛)
- 不追求"取代简历"(简历在,我们做更深一层)
- 不追求"做一个工作申请工具"(这会把用户拉回 LinkedIn 式焦虑)
- 不追求"信息聚合产品的市场份额"(这是 hook,不是 destination)

## 三、产品逻辑(How)

### 3.1 完整闭环逻辑

```
[Hook] 高质量行业时报(用户进来的原因,差异化的"个性化")
   ↓
[第一次互动] AI 问:"我要把时报调得更适合你,告诉我你在做什么"
   ↓
[第一次链接动作] 链接 1 个账号(GitHub / X / Substack 任一)
   ↓
[Magic Moment ≤30 秒] AI 用链接的内容,30 秒内给用户一个
"你自己都没意识到的关于自己的事实" + 一份初版能力地图
   ↓
[第二次留存触发] 时报变得更精准 + 能力地图开始有形 +
AI 提出"再链接什么可以让镜子更准"
   ↓
[周-月活循环] 时报每天读 → AI 标记你看了什么 →
能力地图持续更新 → 用户偶尔回来更新近况 →
AI 写"季度复盘",用户感觉"这产品比我更懂我"
   ↓
[隐性产出] 我们沉淀:跨平台 proof + 阅读行为指纹 +
能力地图 + 学习/兴趣意图 + AI 工作流痕迹
   ↓
[阶段二激活] 用户主动决定:把能力 page 给谁看
   ↓
[阶段三激活] 用户主动开 opportunity 通道
```

**关键洞察**:产品不是"信息聚合"+"个人能力页"两件事拼在一起,而是**一件事**——
信息聚合的输入决定能力页的输出,反过来能力页的输出又决定下一轮信息聚合的精度。
**这是一个 self-reinforcing loop**——这就是 product moat。

### 3.2 产品形态:"能力镜子",不是"身份展示"

**核心比喻**:Strava 之于跑者。

| 别用这些做参考 | 用这些做参考 |
|---------------|-------------|
| LinkedIn(身份展示) | **Strava**(运动数据 + 个人 dashboard + 可选社交) |
| Polywork / Read.cv / Bento(简介聚合) | **23andMe**(为了解自己付费,数据是研究副产品) |
| Substack(reader-first) | **Notion / Linear**(为自己用的 OS) |
| Twitter / X(发声平台) | **Roam / Obsidian / Mem**(第二大脑,反思工具) |
| Indeed / 求职网站 | **Duolingo**(能力提升过程的可视化) |

**产品的"主画面"应该是**:**用户的能力镜子 dashboard**——动态的、AI 维护的、关于
"我是谁、我在变成什么样"的视图。

行业时报是 **contextual sidebar**,是为了让用户每天有理由打开;然后顺便更新自己的能
力地图。

### 3.3 用户动机分层

设计任何产品功能时,问:这个功能在第几层动机?

| 层 | 动机 | 用户感受 | 例子 |
|---|------|--------|------|
| 1 | 被迫 | "我不得不更新简历" | LinkedIn 求职时 |
| 2 | 交易 | "我给数据换服务" | 数据中介 |
| 3 | 自我表达 | "我想被看见" | Substack / X |
| 4 | 身份建构 | "帮我成为更好的自己" | Strava / Duolingo |
| 5 | 职业发展 | "让我变得更值钱" | LinkedIn 想做但做坏了 |

- LinkedIn 把"5"用"1"的体验交付了——把"职业发展"变成"被审判焦虑"
- Polywork / Read.cv 停留在"3"没能升到"4-5"
- 我们要在 **"3+4 的交集"**——自然滑向"5"作为隐性奖励

**产品语言铁律**:
- 用 "你"(对自己说)而不是 "他们"(雇主/同行)
- 用 "看见自己" 而不是 "被看见"
- 用 "能力地图" 而不是 "简历"
- 用 "我在变成什么样" 而不是 "我能匹配什么职位"

### 3.4 关键产品原则(不可动摇)

1. **First-day value 不依赖发布或社交**——单用户就有完整价值,网络效应是奖励
2. **Zero social pressure**——默认私密,用户掌握每一片数据的可见性
3. **AI 是"镜子"不是"麦克风"**——帮你看清自己,不帮你包装自己
4. **任何"对外可见"功能都是用户主动 opt-in**
5. **数据可逆**——用户可以随时删除任何链接、撤回任何公开
6. **B 端兑现永远是 opt-in 且可关闭**,不影响其他功能

## 四、链接账号的动机设计(产品成败的关键)

### 4.1 五层动机模型

让用户链接 GitHub / X / Substack 是整个产品成败的关键动作。设计上分五层:

**Level 1 — 绝对不要做**
"链接这些账号,我们帮你做更好的简历" → LinkedIn 式恐吓,直接让用户警惕"我被采集"。
**永远不要用这个文案。**

**Level 2 — 30 秒 Magic Moment**(必须做,且必须 day 1 做对)
用户链接 GitHub 后 ≤30 秒,产品返回一个**用户自己都没意识到的、关于自己的洞察**。例:
- "你过去 6 个月 commit 最高的语言变了——从 Python 转向 TypeScript,这是技术栈漂移信号"
- "你 Star 的项目里,'AI agent' 主题占比从 2024 Q4 的 8% 涨到 2026 Q1 的 41%——你的兴趣明显在收敛到 agent"
- "你最常合作的 @xxx 和 @yyy 从 2025-09 起没新项目——你的 collaboration 圈层可能需要扩展"

输出**默认完全私密**,只用户看得到。用户感觉"卧槽,这工具有点东西",而不是"我给了
它东西"。

**Level 3 — "再链接一个会更准"的边际激励**
链接 GitHub 后:"想看你的兴趣是否跨领域?链接 X,我会对比 commit 行为和阅读行为。"
——每多链接一个,镜子更准。

类比:Strava 用户主动连接心率手表 + 体重秤 + 睡眠 app,不是 Strava 要求,是用户**为
了让镜子更准而主动连**。

**Level 4 — "我的页面,可选择性公开"的赋能**
当用户已经链接 3-5 个账号、有了能力地图,产品提供:"这是你的能力 page,默认私密。
你可以选择性公开给:朋友 / 同行 / 招聘方 / 完全公开。"

用户体验:**"我有了一个我自己控制的对外接口"**——可以放进 X bio、邮件签名、个人
站、给老板看的能力汇报。

**Level 5 — 隐性 B 端通道(只在用户 opt-in 后开)**
24 个月后,用户已经积累真实能力 page,产品提供 toggle:"是否允许 AI 招聘平台匹配你
fit 的机会?你不会看到广告,你只看到具体 offer(谁、什么职位、薪资范围、技术要求),
你决定回不回应。"

**Opt-in default off**。我们的 B 端商业化全部建立在这一层上,且永远**不影响 Level 2-4**。

### 4.2 Magic Moment 的设计原则

这是 day 1 必须做对的事。原则:

1. **必须是"用户自己都没意识到"的事实**——不能是"你最常用 Python"这种废话
2. **必须基于数据计算,不是 AI 编**——可验证(否则用户会用 AI 测试它的真伪)
3. **必须呈现一个"小惊讶"**——情绪是 "huh, interesting" 不是 "yeah I know"
4. **必须是有 actionable 含义**——不是事实陈述,是"这意味着什么"
5. **必须 30 秒内出**——超过 30 秒用户已经走了

候选 Magic Moment 类型(下一轮调研要 prototype):
- **技术栈漂移**(GitHub commit 分布的时间变化)
- **关注圈层演化**(X follower / following 的领域分布演化)
- **兴趣收敛信号**(GitHub stars / X bookmarks 的 topic 收敛)
- **协作网络分析**(co-author / co-mention 网络)
- **写作主题分布**(Substack / Medium 文章的 topic clusters)
- **学习曲线可视化**(从历史产出推断"能力演化曲线")
- **"你的影响力受众"分析**(谁在引用/转发你的产出)

### 4.3 数据归属和隐私模型(产品 trust 的基石)

**用户必须感知到的事实**:
1. **数据归用户所有,产品方仅托管**——明文声明,UI 持续强化
2. **默认全部私密**——任何"公开"都是用户主动 toggle
3. **可一键导出**(JSON / Markdown,全量、机器可读)
4. **可一键删除**——所有数据从 production + backup 内 30 天彻底清除,有 audit log
5. **B 端通道完全 opt-in**——永远默认关闭,关闭后即时生效
6. **第三方平台数据(GitHub / X)读取后镜像 + 缓存我们自己的版本**,但用户可以
   随时让我们丢弃缓存
7. **不卖广告,不上 AI 训练**——只用于"用户镜子 + 用户授权的 B 端匹配"两个用途

**这是产品 trust 的基石,无论是营销文案、UI 微文案、还是 Terms,都要在 Day 1 一致
对齐**。一旦在阶段一妥协(比如悄悄打开 B 端通道),整个 thesis 死。

## 五、BC 生态的三阶段路径

### 5.1 阶段一:C 端单边价值(0-12 月)

**目标**:在 1-3 个细分人群里做到"周活专业工具",数据沉淀但不变现。

**C 端动作**:
- 用户用产品做个人能力沉淀(私密、AI 辅助反思)
- 每天读个性化时报、每周更新一次自己、每月看一次 AI 季度复盘
- 偶尔主动链接新账号 / 上传新产出

**B 端动作**:
- **完全不开 B 端通道**
- 产品页**不提**任何"被招聘 / 找工作 / 求职"字眼
- 数据沉淀只用来训练自家能力图谱模型

**关键 KPI**:
- 1-10 万 MAU(细分人群定义清晰,小池子可达)
- WAU/MAU ≥ 50%(证明非求职场景下的真实粘性)
- 平均链接账号数 ≥ 3
- 平均 page 公开率(可选指标):0%——故意不引导公开

**风险**:阶段一 12 个月零变现。需要投资人/团队耐心。这点要在 Day 0 跟所有
stakeholder 对齐。

### 5.2 阶段二:C 端可选公开(12-24 月)

**目标**:用户可以选择性公开"能力 page",支持嵌入个人站、X bio,但仍不卖给 B 端。

**新增功能**:
- 能力 page 公开 toggle(默认 off)
- 可定制公开内容(选择性公开哪一片能力)
- 嵌入式 widget(可嵌入 individual 网站、Notion、Substack about page)
- LLM-friendly meta tags(让 ChatGPT / Perplexity / Gemini 能搜到用户公开 page)

**核心策略**:把用户的公开 page 优化为 **"AI-search-indexed"** 的格式——这是 Phase 4
报告里 AI consultant 已经在讨论的 emerging signal。我们 day 1 就 build 这一层,等 LLM
搜索成熟时,我们已经是 default 标准。

**关键 KPI**:
- 30% 用户公开过至少一片能力
- 在 ChatGPT / Perplexity 上搜目标人群名字时,能搜到我们的 page
- 出现 ≥10 个"非招聘场景"成功故事(被合作方/投资人/同行找到)

### 5.3 阶段三:用户主动开通 B 端通道(24-36 月)

**目标**:开 B 端通道,但用 reverse model——不是招聘方主动搜,是用户允许后才能联系。

**Reverse model 设计**:
- B 端付费(订阅 + per-contact)
- 招聘方提"具体机会"(谁、职位、薪资范围、技术要求),不是"广撒网"
- 用户全程审核:看到机会 → 决定回不回应
- 数据使用:仅在用户 opt-in 后,允许我们用其能力数据做匹配

**关键 KPI**:
- 20% 用户开通 opportunity 通道
- B 端 ARR(具体数字基于市场预测,这里不细做)
- 用户拒绝率 / 回应率监控——拒绝率过低或过高都是产品病

## 六、目标用户和切入路径

### 6.1 首批用户画像(基于 v1 Phase 3 综合)

切入顺序(基于 v1 Top 3):

**首切:Marketing 长尾 + transitioning**
- 画像:25-40 岁 in-house marketer,想 transition 到 vibe/Gen marketer
- 痛点:AI 工作流没地方沉淀,LinkedIn 上的 marketing 内容 99% 是 AI slop,他们想被
  区分但没工具
- 可触达 TAM:600K-1M 英语圈
- 渠道:Kyle Poyar Growth Unhinged(84K)、Emily Kramer MKT1、Lenny's、Demand Curve

**同步切:Design 务实派 + AI 拥抱派**
- 画像:UX/product designer @ AI startup,或 motion/AI video designer
- 痛点:Dribbble 死了、portfolio 形态在重塑、AI workflow 没标准展示方式
- 可触达 TAM:50-70 万英语圈
- 渠道:X(Pablo Stanley、Soren Iverson 等)、Behance、Femke's Design Buddies
  Discord

**6-12 月扩张:Consulting 独立/fractional 段**
- 画像:ex-MBB indie、fractional CXO,LinkedIn 自称"fractional"的活跃部分
- 痛点:NDA cage 让没法 show artifact,fractional 增长 55x 但 LinkedIn 千篇一律
- 可触达 TAM:10-15K 真正活跃
- 渠道:Lenny's、Maven、Hampton、Pavilion

### 6.2 不应是首批用户的人

明确说出来,避免错把谁拉进来:

- ❌ **求职中的人**(他们要的是 LinkedIn 替代品,不是能力镜子)
- ❌ **大律所 partners、IB seniors、大医院主任医师**(身份已被外部产品锁定,不可撬)
- ❌ **BigCo HR / IT / Finance 员工**(LinkedIn 婚姻状态,迁移意愿 0)
- ❌ **学生 / fresh grad**(没有"能力沉淀"可镜像)
- ❌ **AI evangelists / Twitter influencers**(已经在自己 substack 上发声,不需要新平台)

## 七、关键假设(必须验证)

这一节是下一轮调研的"靶子"——这些假设有任何一个证伪,整个 v2 thesis 都要重做。

### 7.1 产品端假设

**A1 — Magic Moment 真的能做到**(置信度:中)
LLM + 用户 GitHub 数据,30 秒内能输出"用户自己都没意识到的、有质感的、可验证的"
洞察。如果只能输出"你最常用 Python"这种废话,产品方向死。
**怎么验证**:写一个 prototype,自己 + 10 个朋友试。8/10 反应 "huh, that's
interesting" 才算合格。

**A2 — 行业时报的个性化在 cold start 也有质感**(置信度:中-低)
用户刚来时能力地图数据稀薄,LLM 个性化输出能否明显优于通用 newsletter?
**怎么验证**:模拟 5 个"已用 3 个月"虚拟用户 vs 5 个"刚链接 1 个账号"虚拟用户,
LLM 输出的内容质量对比。

**A3 — 链接账号的 5 层动机模型在真实用户身上成立**(置信度:中)
理论上"Magic Moment → 想链接更多"是 self-reinforcing 的,但实际可能用户链接第一个
后就走人,不 progress。
**怎么验证**:做 prototype 后跟踪用户从链接 1 个到链接 3+ 个的 conversion 路径。

### 7.2 用户端假设

**B1 — 目标人群有"能力沉淀"的真实需求**(置信度:中-高,但来自 desk research)
v1 Phase 3 反复出现"想 sanitized workflow proof""junior pipeline broken""能力没地
方沉淀"等信号,但全是间接证据。
**怎么验证**:与 20 个目标人群(7 marketer / 7 designer / 6 consultant)做 1 小时
深度访谈,问"你最近一次想给别人证明自己能力是什么时候?你做了什么?最难的是什
么?"——听他们自己说,而不是验证我们的假设。

**B2 — 用户愿意 day 1 链接至少 1 个账号**(置信度:中-低)
即使 Magic Moment 设计好,陌生产品要求 OAuth 是有摩擦的。
**怎么验证**:landing page 测试——给 100 个目标人群看"链接 GitHub 看见自己都不知道
的事实",测点击 → OAuth completion 转化率。

**B3 — 用户在阶段一(零兑现)期间会留下来**(置信度:中)
"为自己沉淀"的纯粹动机能否撑 6-12 个月持续打开?Strava 可以,但 Strava 有清晰的"运
动数据 → 健康改善"反馈环。我们的"能力地图 → 职业改善"反馈环要慢很多。
**怎么验证**:阶段一启动后,跟踪 D7 / D30 / D90 留存曲线,与 Notion / Strava 做基
准对比。

### 7.3 市场端假设

**C1 — LinkedIn 5 年内不会推出竞品**(置信度:中)
v1 Phase 4 说 LinkedIn 已经在主动锁定"verified identity"高地。如果它推出"AI
Capability Page"功能,我们的差异化大幅缩水。
**怎么验证**:监控 LinkedIn 的产品发布节奏(每季度看一次 LinkedIn News + Bersin
analysis)。

**C2 — Marketing/Design/Consulting 三人群不会被现有工具完全服务**(置信度:中)
Maven / Reforge / Pavilion 等已经在做"职业发展课程",Lovable / Bolt 在做 portfolio
microsites。这些会不会演化成我们的竞品?
**怎么验证**:深度跟踪这 5 个产品 24 个月路线图,如果它们 pivot 到能力镜子方向,
我们差异化要重做。

**C3 — 阶段一可以在小池子内冷启动**(置信度:中)
Polywork / Read.cv 都死于冷启动,我们能不能用"为自己用,单边价值"避免冷启动陷阱?
**怎么验证**:第一批 100 用户 invite-only,看真实 D30 留存。30% 算合格,50% 算优秀。

### 7.4 商业端假设

**D1 — 阶段一 12 个月零变现可承受**(置信度:取决于资本/团队配置)
12 个月 burn rate × 团队规模 = $X M 资金需求。这是 hard cap。
**怎么验证**:跟决策者(投资人/CFO)对齐"12 个月零变现"是否真的能撑过去,而不是
6 个月就要看 ARR 曲线。

**D2 — 阶段三的 reverse model 能转化用户**(置信度:低)
"用户 opt-in 后才能被联系"模式下,B 端付费意愿是否依然存在?这是反传统的招聘 GTM
模式。
**怎么验证**:阶段二做 5-10 个 B 端付费意向访谈(选 v1 Phase 4 里提到的 SeekOut /
Eightfold 客户群),问"如果用户必须 opt-in 才能被联系,你愿意为单次 contact 付
$X 吗?"

**D3 — 阶段三的数据量真的足以建护城河**(置信度:中)
即使阶段三有 30% 用户开通 opportunity 通道,从规模上能撑得起 B 端 ARR 目标吗?
**怎么验证**:阶段一启动前做数据规模模型——目标 ARR / 单 contact 单价 = 需要多少
active opportunity users / 转化率 = 需要多少 MAU。这个倒推数字如果荒谬,商业模式
要重新设计。

## 八、待调研的问题(下一轮调研清单)

下一轮调研应**聚焦验证第七节的关键假设**,而不是重复 v1 的市场扫描。建议的调研结构:

### 8.1 必答问题(下一轮必须回答)

**Q1 — Magic Moment 的具体形态和可行性**(对应 A1)
- 调研对象:GitHub API、X API(已 paid)、Substack 公开数据的实际可获取范围
- 调研产出:每个数据源能产出哪些类型的 Magic Moment(5-10 个具体候选)+ 实际 prototype 测试反馈

**Q2 — 用户访谈:目标人群的"能力沉淀"真实需求**(对应 B1)
- 调研对象:20 个目标人群(7 marketer + 7 designer + 6 consultant)
- 调研方法:1 小时深度访谈,问"你最近一次想给别人证明自己能力是什么时候?你做了什么?最难的是什么?"
- 调研产出:验证或证伪"能力沉淀需求"是否存在;提取真实痛点描述用于 messaging

**Q3 — Landing page + OAuth 转化率测试**(对应 B2)
- 调研对象:100-500 个目标人群
- 调研方法:landing page 投放(Twitter ads / 行业 newsletter sponsor),测"看到 Magic Moment 卖点 → 点击 → 完成 OAuth"转化率
- 调研产出:这个 funnel 的真实 conversion rate;Magic Moment 文案的 A/B 测试

**Q4 — 行业时报赛道的真实竞争格局**(对应 A2 / C2)
- 调研对象:目标人群所在行业的 newsletter 生态(Marketing Brew、TLDR、Rundown AI、 Growth Unhinged 等的订阅量、付费率、留存数据)
- 调研产出:这个赛道有没有可插入空间;个性化时报能否真的从这些"通用时报"里抢用户

**Q5 — 类似"自我提升 + 数据沉淀"产品的成功路径研究**(对应 B3)
- 调研对象:Strava、Duolingo、23andMe、Whoop、Notion、Roam Research、Obsidian、Mem、Reflect
- 调研产出:这些产品的冷启动路径、D30/D90 留存、变现时间、与本项目可类比的成功/失败模式

### 8.2 重点验证问题(下一轮应该回答)

**Q6 — 平台 API 的实际可用性和限制**(关键技术约束)
- GitHub API rate limit、X API pricing(目前 paid)、Substack 数据 scraping 合规性、LinkedIn 是否允许 OAuth 入口
- 决定哪些 Magic Moment 是技术上可行的

**Q7 — 隐私和数据归属的法律边界**(关键合规约束)
- GDPR、CCPA、各州 AI 法案对"聚合用户多平台数据"的限制
- 阶段三的 B 端 reverse model 在欧美各国是否合规

**Q8 — 已死竞品的具体死因 deep dive**(对应 C3)
- Polywork、Read.cv、Bento、Lunchclub、Cord 这 5 个的具体死因(不止 v1 Phase 4 的概述)
- 找出可以避免的具体陷阱

**Q9 — LinkedIn 24-36 月内的产品路线图预测**(对应 C1)
- 跟踪 LinkedIn Hiring Assistant 后续版本、Verified Applicant 后续动作
- 预测它什么时候可能推出"AI Capability Page"功能

**Q10 — B 端潜在客户的付费意愿和模式**(对应 D2 / D3)
- 与 5-10 个 AI 招聘平台(包括我们自己的现有 B 端产品)做付费意愿访谈
- 验证 reverse model 的可商业化路径

### 8.3 可选探索问题

**Q11 — 用户在产品里"主动表达"vs"被动沉淀"的比例**
**Q12 — 不同行业人群的"能力地图"schema 差异**
**Q13 — AI 生成内容的版权和归属问题**(用户用 AI 做的产出,版权属于谁?)
**Q14 — 数据可移植性的技术架构**(怎么做到用户随时可删除、可导出)

## 九、与 v1 final_report 的对比

### 9.1 哪些 v1 结论被继承(不变)

- ✅ 不能和 LinkedIn 直接抗衡(Phase 2/4 验证)
- ✅ 首批人群选 Marketing / Design / Consulting(Phase 3 验证)
- ✅ B 端兑现不能是核心卖点(CONTEXT.md 约束)
- ✅ HR 不是 C 端用户而是 B 端买家(Phase 3 HR 反共识发现)
- ✅ 阶段一不开 B 端通道(从 final_report 90 天行动建议延续)
- ✅ 用户访谈是必做的下一步(final_report 局限性章节)

### 9.2 哪些 v1 结论被修正(变化)

- 🔄 **产品定位语言修正**:从"AI 时代能力证据层"修正为"职业能力镜子"——更明确
  自我提升语言,远离"职业身份产品"的 LinkedIn-shadow
- 🔄 **Hook 修正**:从"用户主动建 portfolio"修正为"用户来读时报,顺便链接账号"
  ——降低 day 1 门槛
- 🔄 **首日价值修正**:从"持续维护能力 page"修正为"30 秒 Magic Moment"——更快
  抓人
- 🔄 **网络效应处理修正**:从"链出去给 LinkedIn 做补充"修正为"链出去 + 私密为
  默认 + 可选公开"——更强调 trust 维度
- 🔄 **行业时报作用修正**:在 v1 final_report 里未明确,v2 明确为"主 hook,但是
  contextual sidebar 不是 destination"

### 9.3 哪些 v1 局限性在 v2 仍然存在

- 全部假设仍未经用户访谈验证(v1 final_report 已明确,v2 第七节再次强化)
- 欧美数据偏差仍存在(v1 第六节"调研局限性")
- LinkedIn 5 年窗口的具体节奏仍不确定
