# V2 报告 Review 与方向审计

**日期**:2026-05-27  
**对象**:`v2_final_report.md`、`product_thesis_v2.md`、`CONTEXT.md`、V2 Phase 01-07 分析结论  
**目的**:检查 V2 报告是否真实、客观、可执行,并重新对齐第一性目标:打通生态,获取 C 端简历 / 能力 / 技能数据,反哺 B 端 AI 招聘产品。

---

## 0. 最终判断

`v2_final_report.md` 的大方向基本正确,但还不够“业务一阶”。

它已经正确否定了三个危险方向:

1. 不做新的 LinkedIn。
2. 不做纯公开主页 / 作品集社区。
3. 不做后台静默卖数据的 broker。

这些判断和 V2 调研数据一致,尤其是 Read.cv / Polywork / Bento.me 等产品失败、用户不迁移、LinkedIn 仍是默认门面、GitHub / LinkedIn / X / Substack API 与 ToS 限制、EU AI Act / CCPA 合规压力等证据互相吻合。

但是,当前报告的问题是:为了让 C 端产品“讲得通”,它把重心放到了“职业能力镜子”的用户价值上;而从公司第一性目标看,这还不是最终目标。

真正的第一性目的应重新写成:

> 通过一个用户愿意长期使用和授权的 C 端产品,持续获取、更新和结构化 C 端的职业数据、简历数据、能力技能数据、项目证据数据、求职/机会偏好数据,最终补强 B 端 AI 招聘推荐的候选人数据库和匹配质量。

因此,优化方向不是推翻“能力镜子”,而是把它从“产品终点”降级为“数据获取与授权的前台体验”。

更准确的产品定义应是:

> 面向 AI-augmented professionals 的 Career Data Wallet / Ability Passport:用户用它管理自己的职业事实、能力证据和机会偏好;平台在用户授权下把这些数据转化为 B 端可用的高质量候选人信号。

中文内部可以继续叫“职业能力镜子”,但产品团队必须知道:镜子只是用户愿意交数据的理由,不是公司最终要卖的东西。

---

## 1. 第一性目的重述

### 1.1 不是做 C 端工具,而是建设候选人数据生态

`CONTEXT.md` 已经说得很清楚:现有业务是一个面向海外市场的 B 端 AI 招聘产品,依靠大规模简历 / profile 数据为企业做候选人推荐。数据规模越大、质量越高、新鲜度越好,推荐越准,B 端越好卖。

所以,C 端产品的存在理由不是独立做一个 consumer SaaS,而是解决 B 端数据瓶颈。

这个数据瓶颈包括:

1. **规模**:B 端数据库需要更多候选人。
2. **新鲜度**:简历和 LinkedIn 更新慢,很多人 1-2 年才更新一次。
3. **维度**:传统简历只写职位和经历,缺少 AI workflow、项目证据、工具栈、真实产出。
4. **真实性**:AI 时代简历和作品更容易造假,需要来源、时间戳、跨平台一致性。
5. **可挖动性**:B 端最想要 passive candidate,但用户不愿公开求职。

V2 thesis 也已经提出数据护城河公式:

```
数据护城河价值 = 数据维度 × 数据规模 × 数据新鲜度 × 数据真实性
```

这个公式应该成为所有产品判断的最高标准。

### 1.2 C 端产品的核心 KPI 应改写

当前报告中有一些 C 端产品 KPI,如 D30、证据卡片、公开页面、$20/月付费等。这些有用,但不应是最高层指标。

更接近第一性目的的 KPI 应该是:

| 层级 | 核心指标 |
|---|---|
| 数据规模 | 有效候选人 profile 数、可联系候选人数、授权候选人数 |
| 数据新鲜度 | 30 天内更新率、数据源自动同步率、用户主动更新频次 |
| 数据维度 | 每人平均结构化技能数、项目证据数、AI workflow 数、工具栈数 |
| 数据真实性 | OAuth 来源比例、带时间戳证据比例、跨源一致性分数 |
| 可挖动性 | 私密 open-to-opportunity 用户数、机会偏好完整度、回应率 |
| B 端价值 | 推荐命中率提升、B 端点击/收藏/联系率、qualified intro 数 |

C 端留存和付费是手段。最终要看这些行为有没有变成 B 端推荐系统更好的候选人数据。

## 2. 对 `v2_final_report.md` 的真实性与客观性 Review

### 2.1 客观性较强的部分

#### A. “不要做 LinkedIn 替代品”是正确判断

V2 多份调研都指向用户不会迁移,只会叠加。LinkedIn 仍是职业身份的默认门面,但用户会把真实表达、作品、观点和受众放在 X、Substack、GitHub、个人站等地方。

这说明新产品不应挑战 LinkedIn 的默认层,而应寄生在 LinkedIn 之外的证据层。

报告里“用户不会迁移,只会叠加”的判断真实、客观、可执行。

#### B. “纯展示产品会死”是正确判断

Read.cv、Polywork、Bento.me、Path、Lunchclub 等案例说明,没有交易钩子和持续回访理由的职业展示产品很难长期成立。

这证明当前报告否定“公开主页 / 作品集社区”的方向是对的。

#### C. “后台静默反哺 B 端不可行”是正确判断

Phase 07 的约束很硬。LinkedIn 数据抓取不可依赖;GitHub ToS 明确限制把用户数据卖给 recruiter / headhunter / job board;X 成本过高;Substack 无官方 API;EU AI Act 对招聘 AI 风险极高;CCPA 对 sale 的定义很广。

这说明产品不能做“用户连账号,我们后台卖给 B 端”的数据 broker。必须改为 user-initiated / user-authorized。

报告在这一点上判断正确。

#### D. “先切 Marketing,暂缓 Consulting / Legal / Healthcare / Finance”基本正确

Marketing 人群公开承认 AI 工具,有内容 / 工具 / 课程 / 社群付费习惯,可脱敏产出较多,也更容易通过 KOL 触达。

Consulting、Legal、Healthcare、Finance 都有真实需求,但存在 NDA、合规、监管、职业风险和作品不可公开问题。把它们放到第二阶段是稳妥选择。

### 2.2 需要修正的部分

#### A. 报告对“能力镜子”的强调略高于“数据资产”

当前报告把“能力镜子”写成产品主形态,这有利于 C 端理解,但会让团队误以为目标是做一个个人成长工具。

正确理解应是:

> 能力镜子 = 用户愿意持续提交 / 授权数据的前台体验。

它的价值不在于本身能不能卖 $20/月,而在于能否让用户持续给出最新的职业事实和能力证据。

因此,报告应进一步强调“数据对象模型”:用户每一次连接、上传、编辑、生成证据卡,都必须落成可用于 B 端推荐的结构化字段。

#### B. 90 天 MVP 仍偏“展示卡片”,不够偏“简历 / 技能采集”

报告建议 MVP 做 workflow case study builder,方向没错,但第一性目标是拿到 C 端简历和能力技能。

所以 MVP 首屏不应只问“连接账号生成洞察”,还应明确构建四个数据资产:

1. Resume Core:职位、公司、年限、教育、地区、联系方式。
2. Skill Graph:技能、工具、行业、职能、AI 工具栈。
3. Evidence Graph:项目、案例、链接、指标、时间戳、来源。
4. Opportunity Graph:是否开放机会、偏好岗位、薪资、地点、远程、行业。

没有这四个图谱,能力镜子就只是一个内容体验,不能真正反哺 B 端。

#### C. 个人付费不应作为主商业判断

报告提到 $20/月是合理价格锚,这个判断真实,但容易让团队把重点放到 C 端订阅收入。

从第一性目标看,C 端付费不是核心。甚至免费也可以接受,只要用户贡献的数据足够新鲜、真实、有维度,并且能提升 B 端推荐质量。

更合适的商业判断是:

- C 端付费:验证用户价值和留存。
- B 端价值:验证数据是否能改善推荐、触达和转化。

如果两者冲突,优先 B 端数据价值。

#### D. “行业时报”仍然偏弱

Newsletter 调研显示,newsletter 适合作为高信任渠道,但不适合作为大众用户持续贡献职业信息的核心机制。

如果继续保留行业时报,它必须和数据采集绑定:

- 不是“给你看行业新闻”。
- 而是“这些行业变化与你的能力地图有什么关系”。
- 每次阅读后,产品都应该引导用户更新技能、兴趣、目标岗位或证据卡。

否则行业时报只是内容产品,和获取简历/技能数据的目标关系不够直接。

## 3. 应优化到哪个方向

### 3.1 从“职业能力镜子”优化为“职业数据钱包 + 能力护照”

建议方向不是换赛道,而是重命名内部产品模型。

外部用户语言可以是:

> Career Mirror:看见你正在变成什么样。

内部业务语言应是:

> Career Data Wallet / Ability Passport:持续获取、结构化、验证和授权候选人职业数据。

这两个语言不冲突。前者解决 C 端动机,后者服务 B 端数据护城河。

### 3.2 产品闭环应从“内容闭环”改成“数据闭环”

当前报告里的闭环是:

> 连接数据源 → 产生能力洞察 → 生成证据卡 → 用户分享 → 用户回来更新。

建议改成:

> 连接 / 上传 / 导入职业数据 → 系统生成结构化简历和技能图谱 → 用户确认和补全 → 系统生成能力证据和机会偏好 → 用户授权在特定场景使用 → B 端推荐系统反馈哪些字段有价值 → C 端继续补全。

这个闭环更符合第一性目的。

### 3.3 MVP 应优先做“简历和技能采集”而不是“漂亮证据页”

MVP 的第一版应该包含:

1. LinkedIn export / 简历 PDF / 手动简历导入。
2. GitHub / Substack / 个人站 / portfolio URL 连接。
3. AI 自动抽取职位、技能、工具、项目、行业、成果指标。
4. 用户确认并修正。
5. 生成私密能力地图。
6. 生成可分享能力证据页。
7. 机会偏好 toggle。

这里最重要的是第 3 和第 4 步:AI 抽取 + 用户确认。因为只有用户确认过的数据,才有机会成为高质量 B 端数据。

### 3.4 首切人群建议微调

原报告建议首切 AI-augmented marketing operator,这个仍然成立。但目标用户应再细分成两组。

第一组:主动展示型 marketer。  
这类人愿意发布案例、分享证据页,有利于冷启动传播。

第二组:沉默但可挖型 marketer。  
这类人不爱发 LinkedIn,但愿意私密整理能力和开放机会通道。他们对 B 端数据库更有价值。

MVP 不应只服务第一组,否则会陷入 creator 小众市场。真正的数据护城河来自第二组。

## 4. 建议的新产品架构

### 4.1 四个核心数据对象

#### 1. Candidate Profile

包含:

- 姓名
- 邮箱
- 地区
- 当前职位
- 当前公司
- 工作年限
- 教育背景
- 过往经历
- 可联系状态

这是 B 端推荐的基础。

#### 2. Skill Graph

包含:

- 职能技能
- 行业技能
- AI 工具技能
- 软件工具栈
- 语言 / 框架 / 平台
- 熟练度
- 最近使用时间
- 来源证据

这是区别于传统简历的核心增量。

#### 3. Evidence Graph

包含:

- 项目
- 作品
- case study
- 链接
- 时间戳
- 贡献角色
- 指标结果
- AI 参与方式
- 来源平台
- 可信度评分

这是真实性和能力 proof 的核心。

#### 4. Opportunity Graph

包含:

- 是否开放机会
- 想看什么机会
- 不想看什么机会
- 薪资 / rate 区间
- 地点 / 远程偏好
- 行业偏好
- 当前求职敏感度
- 可联系渠道

这是 passive candidate 激活的核心。

### 4.2 C 端界面可以包装成三个页面

1. My Career Mirror:我是谁,我在变成什么样。
2. My Proof:我有哪些可验证能力证据。
3. My Opportunity Control:我愿意让谁、在什么条件下看到我。

### 4.3 B 端接口不能是搜索数据库

至少第一阶段不要给 B 端一个“搜索所有 C 端用户”的后台。

更安全的模式是:

- B 端提交具体岗位 / 项目需求。
- 系统在授权用户中做匹配。
- 用户先看到机会。
- 用户同意后才把 profile 发给 B 端。

这更符合 user-initiated 架构,也更容易规避“数据被卖”的用户感知。

## 5. 对当前报告的建议修改点

建议保留 `v2_final_report.md`,但后续版本应加一个“第一性目的对齐”章节。

应补充以下内容:

1. C 端产品不是最终业务,是候选人数据生态入口。
2. 所有功能必须映射到 Candidate Profile / Skill Graph / Evidence Graph / Opportunity Graph 四类数据资产。
3. 90 天指标要加入数据资产指标,而不只是产品行为指标。
4. MVP 首屏要允许导入简历 / LinkedIn export / portfolio,而不只是连接 GitHub / Substack。
5. “能力镜子”对外可讲,但内部不能忘记目标是获取简历和能力技能数据。
6. B 端闭环要提前设计,哪怕第一阶段不商业化。

## 6. 重新排序后的优先级

### P0:数据采集基础能力

- 简历 PDF / DOCX 导入
- LinkedIn export 导入
- 手动 profile 编辑
- GitHub / 个人站 / RSS 连接
- AI 抽取技能和项目
- 用户确认机制

### P1:能力镜子体验

- 能力地图
- 技能变化
- 证据缺口
- 职业定位变化
- 月度复盘

### P2:证据卡和公开页

- workflow case study
- 公开 proof page
- LinkedIn Featured 导出
- schema.org / JSON-LD

### P3:机会通道

- open-to-opportunity toggle
- 机会偏好
- B 端需求提交
- 用户审核后介绍

### P4:内容 / 行业时报

- 行业雷达
- 能力差距提示
- 证据补全建议

行业时报排到 P4,不是因为没用,而是因为它不是数据护城河的直接来源。

## 7. 90 天验证应重新定义

原报告的 90 天验证偏产品行为。建议改成数据资产验证。

### 7.1 90 天硬指标

| 指标 | 合格线 | 说明 |
|---|---:|---|
| 完成基础 profile | 60 / 100 | 至少包含经历、技能、联系方式 |
| 导入简历或 LinkedIn export | 40 / 100 | 验证简历采集路径 |
| 连接至少 1 个外部证据源 | 35 / 100 | GitHub / Substack / 个人站 / portfolio |
| AI 抽取后用户确认技能 | 50 / 100 | 验证结构化技能可用性 |
| 生成至少 1 个 Evidence card | 30 / 100 | 验证能力 proof |
| 填写机会偏好 | 25 / 100 | 验证 passive candidate 通道 |
| 第 4 周数据更新 | 20 / 100 | 验证新鲜度 |
| B 端认为 profile 有增量价值 | 5 个访谈中 3 个 | 验证商业闭环 |

### 7.2 成功标准

不是“用户喜欢这个产品”,而是:

1. 用户愿意把简历 / profile / 技能数据交给我们。
2. 用户愿意确认 AI 抽取的技能和项目。
3. 用户愿意持续更新。
4. 用户愿意在一定条件下开放给机会方。
5. B 端认为这些数据比 LinkedIn profile 更有用。

---

## 8. 结论

当前 `v2_final_report.md` 真实度和客观性基本合格,方向也比 V1 更清晰。但它仍然偏“C 端产品叙事”,需要进一步拉回“候选人数据生态”这个第一性目标。

最终优化方向:

> 不做职业社交网络。  
> 不做内容平台。  
> 不做纯作品集。  
> 做一个用户愿意使用的职业数据钱包,用能力镜子作为前台体验,用证据卡作为信任层,用机会通道作为 B 端连接层,持续获取 C 端简历、技能、能力证据和机会偏好。

如果只记住一句话:

> 职业能力镜子不是目的,它是让用户持续授权和更新职业数据的交互外壳;真正的资产是结构化、可验证、可授权流动的候选人能力数据。
